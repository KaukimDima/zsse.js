1. Жизненный цикл компоненты
При создании компоненты с помощью класса, компонента имеет несколько встроенных функций, которые называют lifecycle callbacks, которые вызываются в зависимости от того, что происходит с компонентой.

В целом, компонента может быть примонтирована к DOM, может не быть примонтирована, и может получать новые значения через props. Эти процессы и описываются callbacks.

Обязательной функцией является функция render, которая должна возвращать null, string, портал(о нём позднее), boolean или jsx. Остальные — опциональны.
constructor(props)
Конструктор компоненты вызывается монтированием компоненты к DOM. Обязательным является вызывать super(props) перед какой-либо операцией в конструкторе. Конструктор обычно используется для инициализации state компоненты. Если вам не нужно инициализировать состояние от props, то лучше не использовать конструктор.

Задавать состояние в конструкторе нужно не через setState, а через присваивание this.state = {state: 'init'}.

Также можно задать состояние по умолчанию без конструктора: 
class Input extends Component {
 state = {
   value: 'empty value'
 };

 render() {
   const {value} = this.state;
   return <input value={value} />;
 }
}
componentWillMount()
Эта функция вызывается непосредственно перед монтированием и render.
render()
Функция render должна возвращать jsx, string, number, null, boolean или портал, если возвращается boolean значение или null, то компонент не будет отрендерен.
componentDidMount()
Эта функция вызывается сразу же после вызова функции render и монтировании компонента к DOM. Если вам нужно запустить таймеры, получить данные от сервера или получить значения от DOM, например, getBoundingClientRect, эта функция — идеальное место для этого.
shouldComponentUpdate(nextProps, nextState)
Вызывается перед функцией render, и не вызывается в самый первый раз при монтировании. Эта функция сообщает реакту, нужно ли вызывать функцию render для компоненты, когда у нее изменились её props или state. Функция должна вернуть булево значение. По умолчанию, она возвращает true и компонент производит render каждый раз при изменении state или props.

Если props изменились, и shouldComponentUpdate возвращает false методы componentWillUpdate, render и compoentDidUpdate не будут вызваны.

Единственное отличие PureComponent от Component заключается в том, что PureComponent реализует поверхностное сравнение в методе shouldComponentUpdate.
Поверхностное сравнение (shallow comparison) — сравнение чисел, булевых значенией и строк по значению, а массивов и объектов по ссылке.
При таком сравнении могут возникнуть проблемы, если вы используете мутации. Например, присваивание новых свойств объекту, или добавление в массив новых значений с помощью функций изменяющих массив. Если передать такой объект или массив как props, затем сделать мутацию, компонент не увидит изменений, так как ссылка останется прежней. Поэтому в мире реакт стараются избегать мутаций и создавать новые объекты и массивы при их изменении.
componentWillReceiveProps(nextProps)
Этот метод вызывается каждый раз, когда компонент получает новые props. В этом методе, как правило, обновляют значения state, которые зависят от props.
componentWillUpdate(nextProps, nextState)
Этот метод вызывается непосредственно перед обновлением компоненты и вызовом render.

В этом методе нельзя вызывать setState.

Этот метод позволяет сделать подготовки перед обновлением, например, посчитать какие локальные переменные класса.

componentDidUpdate(prevProps, prevState)
Вызывается сразу после render. Не вызывается после первого вызова render.
Этот метод используется для того, чтобы изменять DOM в зависимости от props компоненты.
componentWillUnmount()
Этот метод вызывается перед тем, как компонент отмонтируется от DOM и будет уничтожен. В этом методе отключают eventListener, если они есть и отменяют сетевые запросы.
componentDidCatch(error, info)
Если в одном из детей компоненты произошла ошибка, будет вызван этот метод. С помощью него можно вызвать setState и указать ошибку в состоянии компоненты, чтобы ее можно было отразить в ui в методе render.
2. Порядок вызовов методов компоненты
1
Монтирование компоненты
Компонента монтируется, если она добавляется компонентом родителем в jsx, или если она указана в методе ReactDOM.render.

Методы вызываются в следующем порядке:

  • constructor
  • componentWillMount
  • render
  • componentDidMount
2
Обновление компоненты
Этот порядок вызывается, когда компонента получает новые props.

  • compoentWillReceiveProps
  • shouldComponentUpdate
  • componentWillUpdate
  • render
  • componentDidUpdate

3
Удаление компоненты
  • componentWillUnmoun

3. Дополнительные методы
forceUpdate()
По умолчанию, компонент вызывает метод render только если он получает новые props, меняет state или монтируется. Есть возможность вызвать в ручную метод render, с помощью forceUpdate.

В нормальной ситуации не стоит вызывать метод, а когда покажется что стоит, нужно подумать еще 7 раз.
defaultProps
Это переменная класса, для того чтобы задать props по-умолчанию. Желательно задать все props, от которых зависит render компоненты.
displayName
Это переменная класса нужна только при разработке и дебагинге. Помогает увидеть, где точно произошла ошибка в стеке ошибок.
4. События в React и их обработка
В реакте используются synthetic event. Это события, которые обладают такими же свойствами, как и обычные события браузера, отличаются в деталях и созданы для того, чтобы во всех браузерах был единый интерфейс взаимодействия с событиями. Для работы с событиями, поступающими от тега или компонента, необходимо использовать атрибут компонента onChange. На самом деле их больше, но пока для примеров будем использовать только onChange.
Пример:
class Form extends React.Component {
 state = {
   value: ''
 };
 handleOnChange = event => {
   this.setState({value: event.target.value});
 };
 render() {
   return (
     <div>
       <p>{this.state.value}</p>
       <input value={this.state.value} onChange={this.handleOnChange} />
     </div>
   );
 }
}
В данном случае компонент Form выводит параграф с содержим state, а также input, значением которого является значение state, и обработчик событий onChange. Обратите внимание, мы передаем в обработчик событий метод нашего класса handleOnChange. Когда мы передаём функцию, по сути мы делаем передачу ссылки на функцию. И если потом вызвать эту функцию, контекст this потеряется, если не предпринять дополнительных мер. Раньше, при разработке на реакте, все методы, предназначенные для передачи в обработчики событий, оборачивали в контекст с помощью функции bind, переданная функция знала, что this указывает на экземпляр компоненты. В новом синтаксисе js, для этих задач служат стрелочные методы js. Метод handleOnChange является стрелочным методом, такой способ указания метода класса делает автоматическую обёртку и в случае передачи такого метода this в нём всегда будет указывать на экземпляр класса.

Метод handleOnChange получает синтетическое событие, с помощью taget.value получаем значение input, записываем в state, и это отражается в параграфе и самом input.

Помимо onChange существуют методы onInput для получения событий input, onSubmit для тегов form, onClick, onFocus для событий получения фокуса и onBlur для событий удаления фокуса. Это список обработчиков событий для мышки:

onClick, onContextMenu, onDoubleClick, onDrag, onDragEnd, onDragEnter, onDragExit, onDragLeave, onDragOver, onDragStart, onDrop, onMouseDown, onMouseEnter, onMouseLeave, onMouseMove, onMouseOut, onMouseOver, onMouseUp.

Все обработчики событий получают синтетическое событие, если нужно остановить баблинг события, можно воспользоваться event.preventDefault() иди event.stopPropagation().

Если нужно передать событие, и воспользоваться им в асинхронном коде, в следующих тиках — необходимо вызвать метод event.persist() чтобы реакт отдал вам это событие. Реакт использует объекты событий много раз, без удаления их из памяти, для ускорения работы, и метод event.persist сообщит реакту, что это событие не нужно очищать и переиспользовать.
4. Композиция компонент
Бывают ситуации, когда вам нужно передать данные из одной компоненты, в другую компоненту, находящуюся рядом. Давайте рассмотрим способы общения компонент, находящихся в близком родстве друг к другу.
1
От родителя к ребенку
Это самый простой вид связи, и помимо передачи значений через props, у нас есть еще один метод, instace methods, то есть встроенные методы ребёнка.

При передачи данных через props, мы просто передаём нужные данные нашему ребенку. Если данные в родителе меняются, мы меняем эти данные в props и ребенок получает себе новые значения.

При использовании встроенных методов, нужно использовать атрибут компонента ref. Ref будет изучен в следующей части, но давайте пока коротко посмотрим, как это работает. 
class Child extends React.Component {
 childMethod() {
   return 'child method called';
 }	

 render() {
   return <p>Child</p>;
 }
}

class Parent extends React.Component {
 componentDidMount() {
   console.log(
     this.child.childMethod() // child method called
   );
 }
 render() {
   return <Child ref={component => (this.child = component)} />;
 }
}
ref — это специальный атрибут, который вызывается, когда тег попадает в DOM. ref ожидает передачи функции. При вызове этой функции, в нее передаётся ссылка на тег. Если это простой тег, то ссылка указывается на объект в DOM, если же это компонент, то ссылка будет указывать на экземпляр компоненты. Реакт примонтировал нашего родителя. Это значит, что все дети тоже уже примонтированы, и мы можем быть уверены в том, что переменная класса child содержит ссылку на экземпляр, которую присвоили в функции, которую передали в ref. Таким образом, мы можем вызвать внутренний метод компоненты, и передать данные, если в этом возникает потребность.
2
От ребенка к родителю
Такой способ общения компонент также имеет 2 простых способа.

Первый способ — это передача callback-функций. Обработчики событий являются примерами этого способа, например: 
class Child extends React.Component {
 render() {
   return <button onClick={this.props.onClick}>Click me</button>;
 }
}
class Parent extends React.Component {
 handleClick() {
   console.log('child clicked');
 }
 render() {
   return <Child onClick={this.handleClick} />;
 }
}
В этом случае ребенок вызывает функцию, которую в него передал родитель в тот момент, когда происходит клик на элемент ребенка.

Второй способ называется event bubbling, вот пример: 
const Child = () => <button>Click me</button>;

class Parent extends React.Component {
 handleClick() {
   console.log('child clicked');
 }
 render() {
   retrun(
     <div onClick={this.handleClick}>
       <Child />
     </div>
   );
 }
}
Тут происходит ловля события, которое происходит на элементе ребенка, но событие, которое порождается ребенком, будет поймано уже у родителя.
3
От ребенка ребенку 
class Parent extends React.Component {
 ...
 render () {
   return (
     <div>
       <Child1
         child1prop={this.state.child2prop}
         callbackChild1={this.handleCallbackChild1}
       />
       <Child2
       child2prop={this.state.child1prop}
       callbackChild2={this.handleCallbackChild2}
     />
     </div>
   )
 }
}
Этот пример — комбинация прежних 2. В состоянии хранится 2 переменные, которые мы передаём 2 детям, калбэки каждого ребенка меняют переменные, которые передаются другому ребёнку.
4
Более сложные взаимодействия
Существуют ситуации, когда нам нужно передать одинаковые данные в совершенно разных местах приложения. В таких случаях используют уже сторонние решения, такие как redux/mobx. Такие решения берут на себя функцию хранения состояния приложения, и предоставляют удобный способ получения нужных данных в разных частях вашего приложения.
5. Списки и keys
В случае, когда нужно вернуть список компонент и используется метод map, необходимо использовать дополнительный атрибут у детей — key. 
const TODOS = [
 { id: 1,
   text: 'помыть посуду' },
 { id: 2,
   text: 'купить продукты' }
];

const List = () => (
 <div>{TODOS.map(todo =>
   <p key={todo.id}>{todo.text}</p>)
 }</div>
);
В данном случае, в key передаётся уникальный id. Каждый раз в key следует передавать уникальный идентификатор. Если нет id, в данном случае можно было использовать text.

Key существует, чтобы помочь реакту понять, какой именно элемент изменился. Помощь присутствует на этапе обновления DOM. В случае, если текст одного из TODO поменялся, после нового вызова компоненты List вернется новый слепок jsx, который произведет слепок virtual DOM, и на этапе сравнения слепков реакт поймет, что реально следует обновить в реальном DOM.
